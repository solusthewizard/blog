Discriminated unions are the pattern you're missing

Most developers reach for optional fields when they should be modelling state. A look at how discriminated unions remove entire classes of bugs from your codebase.

JM
Jamie Moore
Apr 23, 2026
9 min read

There's a class of bug that appears constantly in TypeScript codebases. It looks something like this: a function receives an object, checks one field to determine what kind of thing it is, and then accesses another field that may or may not exist. At runtime, it explodes. The type checker didn't catch it.

The root cause, almost always, is that the developer reached for optional fields when what they actually needed was a discriminated union.

The problem with optional fields

Imagine you're modelling an API response. The request either succeeds (and returns data) or fails (and returns an error message). A common first attempt looks like this:

type ApiResponse = {
  success: boolean;
  data?: User;
  error?: string;
};

This feels fine. But the type says nothing about which combinations are valid. Is it possible for data and error to both be defined? TypeScript has no idea. You'll need to write defensive checks everywhere, and the compiler won't help you if you forget one.

Enter discriminated unions

A discriminated union is a union type where each member has a common field — the discriminant — with a unique literal value. TypeScript uses that field to narrow the type automatically.

type ApiResponse =
  | { success: true; data: User }
  | { success: false; error: string };

function handle(response: ApiResponse) {
  if (response.success) {
    // TypeScript knows: response.data exists here
    console.log(response.data.name);
  } else {
    // TypeScript knows: response.error exists here
    console.log(response.error);
  }
}

Now the type system enforces the constraint. data and error can never appear together. You don't need to defensively check for undefined — TypeScript narrows the type for you based on the discriminant.

Real-world patterns

Discriminated unions shine brightest when modelling state machines — anywhere something can be in one of several mutually exclusive states. Loading states are a perfect example:

type FetchState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; error: Error };
Make impossible states impossible — Elm community maxim, applicable everywhere.

With this type, you can't accidentally access data while in the loading state. The compiler simply won't let you. That's an entire category of bugs eliminated at the type level, before you even run the code.

Exhaustive checking

Another benefit: you can use never to ensure you've handled every case:

function assertUnreachable(x: never): never {
  throw new Error('Unhandled case: ' + x);
}

function render(state: FetchState<User>) {
  switch (state.status) {
    case 'idle': return <Idle />;
    case 'loading': return <Spinner />;
    case 'success': return <Profile user={state.data} />;
    case 'error': return <Error message={state.error.message} />;
    default: return assertUnreachable(state);
  }
}

If you later add a new state to the union and forget to update this switch, TypeScript will error at compile time. That's the kind of safety net that pays for itself the first time you refactor.

The takeaway

Next time you find yourself writing field?: SomeType, pause and ask: is this field optional because the data is genuinely optional, or because you're trying to model a state that simply doesn't exist in certain conditions? If it's the latter, a discriminated union will serve you far better.

The type system is there to help. Let it do its job.
